# ProjectWeatherCloth1 기술 면접 대비 가이드

본 프로젝트의 설계 및 구현 과정을 바탕으로 한 10가지 핵심 면접 질문과 모범 답변입니다.

## Q1. 추천 로직을 단순 조건문이 아닌 점수 기반(Score-based)으로 설계한 이유는?
**A**: 단순 조건문은 기상 변수가 늘어날수록 복잡도가 기하급수적으로 증가하며 유지보수가 어렵습니다. 점수 기반 시스템(CLO 지수 활용)을 통해 기온, 습도, 풍속 등 다양한 독립 변수를 하나의 '필요 보온 지수'로 수렴시키고, 이에 가장 근접한 의류 조합을 선택하는 방식으로 설계하여 확장성과 논리적 명확성을 확보했습니다.

## Q2. FastAPI를 사용하여 백엔드를 구축한 이유는?
**A**: FastAPI는 Python의 비동기(ASGI) 지원을 통해 빠른 성능을 제공하며, Pydantic을 이용한 자동 테이터 검증 및 Swagger UI를 통한 문서화가 강력합니다. 이는 협업 효율성을 높이고 실무 수준의 API 개발 환경을 제공하기 때문입니다.

## Q3. 데이터베이스 추상화(Database Abstraction)를 위해 어떤 기법을 썼나요?
**A**: SQLAlchemy ORM을 사용하여 비즈니스 로직과 물리적 DB를 분리했습니다. 이를 통해 로컬 환경에서는 SQLite를 사용하고, 운영 환경에서는 코드 수정 없이 환경 변수 설정만으로 PostgreSQL 등으로 즉시 교체할 수 있는 유연성을 확보했습니다.

## Q4. 사용자가 늘어날 경우 서버의 병목 현상을 어떻게 해결하겠습니까?
**A**: 1) 실시간 날씨 데이터는 Redis 같은 캐싱 레이어를 두어 중복 API 호출을 방지하겠습니다. 2) 추천 연산이 복잡해질 경우 Celery 등의 비동기 작업 큐를 활용하여 응답 시간을 단축하겠습니다. 3) DB 인덱싱을 최적화하고 읽기 복제(Read Replica)를 고려하겠습니다.

## Q5. API Key 보안은 어떻게 관리하고 있나요?
**A**: 소스 코드에 키를 직접 노출하지 않고 `.env` 파일과 시스템 환경 변수를 통해 관리합니다. 또한 `pydantic-settings`를 사용하여 설정값을 검증하며, 실서비스에서는 AWS Secrets Manager 같은 전문 관리 도구를 연동할 계획입니다.

## Q6. SRP(단일 책임 원칙) 관점에서 코드를 어떻게 리팩토링했나요?
**A**: 기존에 하나의 파일에 섞여 있던 날씨 수집(Weather), 데이터 저장(User), 추천 연산(Recommend) 로직을 각각의 Service 클래스로 분리했습니다. 이를 통해 각 모듈이 하나의 책임만 가지게 하여 테스트와 재사용이 용이하도록 개선했습니다.

## Q7. '체감 온도' 계산에 반영된 요소들은 무엇인가요?
**A**: 기본적인 기온 외에도 풍속에 의한 열 손실, 습도에 의한 증발 억제 효과를 반영했습니다. 또한 사용자의 '개인 보정값(민감도)'과 '건강 데이터'를 가중치로 더해 최종적으로 사용자 맞춤형 보정 체감 온도를 도출했습니다.

## Q8. 테스트 환경에서 외부 API 호출 실패에 어떻게 대응했나요?
**A**: `Simulation Mode`를 구현하여 실제 API 연결이 불가능하거나 키가 없는 상황에서도 시스템이 정상적으로 동작하도록 했습니다. 이는 개발 생산성을 높이고 CI/CD 파이프라인에서의 테스트 안정성을 보장합니다.

## Q9. 프론트엔드와 백엔드를 분리해서 얻는 이점이 무엇인가요?
**A**: 기술 스택의 독립성을 확보할 수 있습니다. 예를 들어 추후 웹 프론트엔드를 React나 모바일 앱으로 교체하더라도, 기존에 구축된 FastAPI 백엔드 API를 그대로 재사용할 수 있어 시스템의 확장성과 재사용성이 극대화됩니다.

## Q10. 이 프로젝트에서 가장 치명적인 설계 결함이 있다면 무엇이며, 어떻게 보완할 수 있을까요?
**A**: 현재 추천 로직이 단일 아이템 점수 기반이라 '패션(코디)' 관점의 조화로움이 부족할 수 있습니다. 이를 보완하기 위해 'TPO(상황)' 태그나 '색상 조화' 규칙을 추천 알고리즘에 추가 반영하여 실용성을 더 높일 수 있습니다.
